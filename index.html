Parse 停止运营的看法

###Parse作为后台终究是一个最差方案
原文翻译自Jameson Quave的Your Parse backend was always a bad idea.

可能你还没听说Parse，这个前不久刚被Facebook收购的后台即服务（BaaS），现在Facebook正准备关闭这项服务。许多开发者听到这个消息不免赶到些许失落，甚至有的感觉被Facebook出卖了。我发布了这样的一条推特，这是一些开发者在Parse关闭前的主页上的留言，差不多是这样的：

我不必添加着重符号，它已经被很好的标识出来：众多的开发者“信任我们”. 你可以从他们图片上所表达的内容上能够感受到，为什么开发者有如此强烈的被背叛的感觉。为何还会有人在知道任何一项技术都可能面临着随时被关闭的风险而继续使用Facebook提供的诸如React Native， React JS， HHVM， Relay这些技术？

诚然，会有一些开源机构和团体可以接管这些服务，但是开源项目所需要维护者和企业资助者将会是一笔巨大的投入。Facebook已经用事实证明它是值得我们信任的，这对于在过去一直使用Facebook提供的API或者使用三方社交媒体API的开发者来讲并不值得惊讶。我稍候会讨论这个问题，但是让我们换个话题，来认识一下另一个被大家忽视的问题–Twitter以及着重的来谈谈Twitter 的 Fabric服务，这个目前拥有crash分析并且通过Felix Krause集成了众多令人惊叹的功能。

但是为了了解Twitter过去如何对待它的开发团队，我们应该谈谈一个叫做Meerkat的小应用。因为和这个故事有相似的经历，我们之后会再回过头来讨论Parse和FaceBook，所以请暂时容忍我一下。

####Meerkat

简略的背景：我生活在Austin，这就意味着每年的SXSW大会我都会对一些之后会得到巨大发展的新兴的项目有一个近距离的了解。 Twitter, Foursquare, GameSalad, 甚至于 Four-Hour Work Week 这样的公司都会在大会上亮相。这是一些已经获得一些成功的例子，但是每年都会有更多的瞪大了眼睛的创始人出现在Austin来展示他们的业务并希望借助大会上的人们能使项目得到推广。2015年的SXSW大会上毫无争议的胜者就是Meerkat。

Meerkat从根本上来说是一个实况直播的P2P平台，提供服务给那些直接由他们的iPhone向其他用户直播的人，而且它在SXSW上掀起了一阵风暴。去年在Austin的SXSW大会上行走一番你会发现Meerkat的广告T恤到处可见。每个人都在流媒体直播音乐，SXSW会议，他们的午餐，甚至于任何他们正在做的事情。但是，突然间…它停止运营了，官方给出了一个非常具体的原因：

在Meerkat当前的运营情况下，Twitter突然收回了API访问权限以及一个两小时的通知

如果你熟悉iOS App Store的流程，你就会了解这是一个什么样的问题，试想Meerkat的使用者们无论如何可以重写整个app而不依赖于Twitter的API，但他们却不能使这个app通过审核并在两个小时内上架到AppStore，实际上这项工作所花费的时间至少是3周这样。

At the height of Meerkat’s launch, Twitter yanked API access right out from under their feet with only 2 hours notice.

于我个人而言，我并没有赶到出乎意料，但是很多人想知道为什么Twitter会采取这样的行动，难道他们违反了API使用条款？或者他们那里不合法？

然而，事实并不是这样…

事实上它揭示了Twitter决定遏制住这个基于Twitter的最成功的app在未来几年的发展的原因正是他们自身也有在这个渠道中的竞争对手，叫做Periscope，这个我不愿给出连接的服务。

所以在这里，Twitter再一次地博取了开发团体的信任。令人扼腕…

####Parse

看吧，说过我们会回到Parse的话题。

Twitter和Facebook的行为从根本上可以说是同一个代表社交媒体的硬币的两面。对于我而言，相信他们中的任何一方都会有相同的教育意义。当我思考到一个科技公司将会在未来的道路上如何发展，我会想起这个信条 “一切朝钱看”。这通常会告诉你一个大型的公司在未来如何运转良好，尤其是那些面向公众的企业（FB和TWTR都是面向公众的）。美国公共企业对其股东的季度财报报告，有时会对他们的客户及合作伙伴造成伤害。当我注意到Parse的”免费分层“的巨大门槛时，着实令我有些担忧。对我而言，99%甚至更多的app将不能顺利通过这个门槛，如果他们通过了这个门槛也将付出几分的代价。当这个项目变成Facebook的开发者工具的时候我们不禁会问，他们的动机是什么尤其是在后端即服务方向？我认为答案很简单：他们想要你的数据，但事实证明这并没什么用处。 所以他们不再让Parse浪费他们的时间。面向众多的开发者通过提够免费的后端服务并不能给他们带来利润；所以他们叫停了这项服务。用Facebook官方的话来说就是：

很荣幸我们为大家创造伟大app提供了帮助，但是我们需要把资源用在其他地方。

翻译过来讲就是：你们没有让我们赚到钱。

Facebook从广告中产生了它的底线，就像Twitter，像Google这个超越了苹果公司成为了当今世界最有价值的公司。事实上，只有一个供应商不把广告作为其营收的主要来源，它就是苹果。

####一切朝钱看

今天的早间时候我发布了这样一条推特：

诚然，你确实不能盲目的信任这些社交媒体公司提供的后端服务。你必须找到以赚钱为目的这种动机的主体。如果他们的动机不是为你提供优质的服务来满足他们的基本营收的，那么让它坚持很长时间是不可能的。如果你是创始人或者CEO，这也正是一个分析你自己所从事的业务的好思路。你和任何人一起工作的时候你必须要确认他们的金钱方面的动机要和你的吻合，否则终究会打破你们的合作。这适用于员工， 联合创始人，合作伙伴，以及供应商。

####Facebook提供的接口

在Facebook提供API的早期，你可以很方便的获取到用户的联系人。这也正是为何从Facebook游戏以及Farmville的崛起中产生了大量的垃圾邮件。但是Facebook判断他们不希望这样，所以他们收回了这项权限，这对许多应用造成了损害，并且影响了Zynga的股票，可以仔细的看下对Zynga股票所产生的影响。

如果你的业务取决于一个应用，那么你的后端则是一个极其重要的而且你必须绝对控制的业务集合。 像Salesforce以及其他基于云服务的企业大概花费了十年时间来探索出走向一个大型企业的道路，即使到今天，他们中的大多数仍在使用在线托管版本的软件。究其原因就在于一个运行良好的业务你必须把一切都视为关键任务。这会增加维护成本以及初始化部署的成本，但是没有这些控制你的业务数据将被一些突发奇想的可疑人挖掘来服务于广告。这难道不是你自己控制服务器的原因？你对Facebook、Twitter和Google的信任又能价值几何？

####构建自己的后端

为你的应用提供稳定的后端服务的不二法门就是自己控制后端。我知道这听起来有些困难，但是这对于Ruby和NodeJS来讲做一个简单的API来驱动你的应用并不是什么难事儿。坦率的讲，Parse后端通过自身的基于javascript事件的机制，并不是和快速的写一个NodeJS应用、为一些简单的API抓取一些节点模块、通过MongoDB数据库来打包所有东西有太大的差别。如果这听起来真的有些困难，那么花费一些时间来学习一下在线教程吧，你就会了解到所有这些自己做起来也并不是太困难。或者你可以雇佣我的公司来做这例行的无休止的嵌入的工作。

如果你真的雇佣了一个供应商来构建你自己的后端，确保你能得到源代码，以及在任何服务器上加载的必要工具。Docker是一个不错的方法来把所有应用所需要的环境打包起来的工具，而且像Heroku这样的服务会是发布Rails应用变得简单。

援引

Temporary Databases

When the name of the database file handed to sqlite3_open() or to ATTACH is an empty string, then a new temporary file is created to hold the database.

rc = sqlite3_open(“”, &db);
ATTACH DATABASE ‘’ AS aux2;
A different temporary file is created each time, so that just like as with the special “:memory:” string, two database connections to temporary databases each have their own private database. Temporary databases are automatically deleted when the connection that created them closes.

Even though a disk file is allocated for each temporary database, in practice the temporary database usually resides in the in-memory pager cache and hence is very little difference between a pure in-memory database created by “:memory:” and a temporary database created by an empty filename. The sole difference is that a “:memory:” database must remain in memory at all times whereas parts of a temporary database might be flushed to disk if database becomes large or if SQLite comes under memory pressure.

PS : 临时数据库是存储在内存的页缓存中，如果数据库很小，那么和驻留在内存中的数据库没有差别，但是当数据库体积随之庞大，临时数据库会呗写入磁盘。

众所周知，Core Data 是苹果内置关于用户数据持久化的解决方案，无论在 iOS 平台还是 OS X 平台，它都是通用的。尽管苹果一直秉持着最大化的使编程简单的宗旨，但是 Core Data 上手对于有经验的开发者也不是一件易事。

即便是你知道如何使用 Core Data ，很简单的日常任务，在使用了 Core Data 之后会变得笨重而繁琐。 好消息是，MagicalPanda开源了一个针对 Core Data 的三方库 – MagicalRecord。 MagicalRecord 提供了便利的方法来创建那些使用Core Data 所必须的代码，诸如对 Core Data 的设置、查询、更新。它的设计灵感来源于德高望重的 Active Record 设计模式。

###问题的来源：

问题描述完毕。

###处理

使用过CocoaPods的朋友知道，在CocoaPods 中添加一个库就是在Podfile 中添加一条记录，并执行 pod install 命令即可。但是在当前（2015-09）CocoaPods 版本稳定版本下 0.38.2 下CocoaPods 集成 AMapSearch 库会出现如下报错

Errno::ENOENT - No such file or directory AMapSearch/AMapSearchKit.framework/Resources

在github上有相关的解决办法，并且有大神表示这是一个CocoaPods版本的bug，在之后的版本会消除。

总结两种解决办法

  1. 删除报错信息中的指定目录或者文件，然后执行 pod install即可；
  2. 手动删除 workspace 下 Pods 中相关的文件，然后执行pod install.

关于手动删除Pods中的三方库

pod install 中出现的错误信息，我们发现一个Pod的集成过程中有一步是向相关目录和文件中写入内容的。所以，删除引用可以从删除这些目录和文件中的三方库信息开始。

  1. Pods 目录下 Manifest.lock 文件中删除AMapSearch行
  2. Pods 目录下 删除 AMapSearch 目录，这个目录下是AMapSearch的framework文件。
  3. Pods 目录下 Headers-> Public 下删除 AMapSearch 目录及其文件
  4. Pods 目录下 Target Support Files 目录中 AMapSearch目录及其文件。

过程中可能会一并出现一些关于版本控制的错误，具体的表现是在Xcode中有 文件找不到，但实际上这些版本控制下的文件是与当前的工程无关的，清除的方法是在 Terminal 中进入相关文件的上层目录执行 *svn delete \.

MVVM初体验

MVVM是一种软件设计中的架构模式。目前在MVC诟病下逐渐由前端过度到移动端的软件设计中。

组成

Model

代表数据状态或者数据访问层的内容。

View

与在MVC和MVP设计模式中一样，View即用户界面。

View Model

ViewModel是View的抽象，其中包含的是共用属性和命令。与MVC模式中的controller、MVP模式中的Presenter均不同。MVVM拥有一个粘合剂。在ViewModel中，这个粘合剂协调View和数据。ViewModel被描述为数据模型中的数据状态。

MVVM是一种编程范式而并不会因此限制你的编程思想。实现技术也有许多种。

具体的实现：

  • View

在实际使用中就是一个ViewController，基于MVVM的基本思想，在ViewController中我们处理的仅仅是用户事件的监听和主要业务逻辑，界面逻辑的处理，但并不会涉及具体的业务逻辑。例如，在登录页面中，View层次负责用户输入数据的验证、登录请求的发起等业务逻辑， 但是实现这些技术的细节，并不在View中实现。

  • ViewModel

这是一个基于 NSObject 的专门处理View页面产生的业务逻辑和界面逻辑实现的类，这个类仅由相应的View调用（由于其功能特殊性，仅针对某个View的业务逻辑进行设计和实现）。

  • Model

Model在本例中具体来讲就是一个用户模型，包含用户的姓名、电话、等属性，但是对于此Model，仅由ViewModel去更新，读取，修改，删除。同样也是基于 View 不负责具体的业务逻辑的原则。

NSURLSession

此API用来通过HTTP协议下载内容，提供了大量的代理方法来提供认证功能，使应用能够在后台或者当应用挂起的时候下载任务。

通过NSURLSession的API，应用创建了一系列的会话，每个都协调着一组相关的数据传输任务。比如，如果你在实现一个浏览器，那么应用会为每个标签或者窗口创建一个sesssion.每个session都会添加一系列的任务，每个任务都代表向一个特定的URL的网络请求。

一个会话的行为有一个设置对象决定，因为这里有三种类型的设置对象，所以相应的就有三种session对象。a.默认的与NSURLConnection很像，代表着session对象不缓存任何内容；b.download session把结果存储到一个文件中，并且当应用挂起、退出、甚至崩溃的时候持续性传输。

通过这些session对象，你可以调度三种类型的任务，数据存储在内存中的，下载到磁盘上的，将磁盘上的文件上传并接收响应的任务。

与其他网络API类似，NSURLSession也是异步的，它的数据返回有两种方式，当然，这取决于你调用了哪个方法:

  • 请求完成时的block，请求成功或者失败的时候通过block把数据返回到应用。
  • 接收到数据的时候调用自定义的代理方法。
  • 下载文件的时候调用自定义的代理方法。
