MVVM初体验

MVVM是一种软件设计中的架构模式。目前在MVC诟病下逐渐由前端过度到移动端的软件设计中。

组成

Model

代表数据状态或者数据访问层的内容。

View

与在MVC和MVP设计模式中一样，View即用户界面。

View Model

ViewModel是View的抽象，其中包含的是共用属性和命令。与MVC模式中的controller、MVP模式中的Presenter均不同。MVVM拥有一个粘合剂。在ViewModel中，这个粘合剂协调View和数据。ViewModel被描述为数据模型中的数据状态。

MVVM是一种编程范式而并不会因此限制你的编程思想。实现技术也有许多种。

具体的实现：

  • View

在实际使用中就是一个ViewController，基于MVVM的基本思想，在ViewController中我们处理的仅仅是用户事件的监听和主要业务逻辑，界面逻辑的处理，但并不会涉及具体的业务逻辑。例如，在登录页面中，View层次负责用户输入数据的验证、登录请求的发起等业务逻辑， 但是实现这些技术的细节，并不在View中实现。

  • ViewModel

这是一个基于 NSObject 的专门处理View页面产生的业务逻辑和界面逻辑实现的类，这个类仅由相应的View调用（由于其功能特殊性，仅针对某个View的业务逻辑进行设计和实现）。

  • Model

Model在本例中具体来讲就是一个用户模型，包含用户的姓名、电话、等属性，但是对于此Model，仅由ViewModel去更新，读取，修改，删除。同样也是基于 View 不负责具体的业务逻辑的原则。

  • 类之间的通讯

View发起一个事件之后，由ViewModel处理事件并通过带参的Block回调给View，再由View显示给用户结果。

AFNetwork文档学习

准备在项目中使用AFNetworking作为网络请求库，使用之前多看看，赶感觉生活更美好

##NSURLConnection

主要功能包含请求创建、响应结果序列化、网络状态检测、安全性、所有请求操作的管理。

  • 子类化注意事项

对于 iOS7 以后，如果有大量的针对网络请求的鼓励使用 AFHTTPSessionManger的子类。这个子类应该提供单例类方法，这样关于请求的认证和一些其他的设置就可以在项目中全局化共享。

  • 需要重写的方法

为了改变所有请求的功能，需要重载 HTTPRequestOperationWithRequest:success:failure方法。

  • 监测网络状况

网络状况的改变可以通过 reachabilityManager 属性监测，这样可以阻塞或者挂起某些请求。

##NSURLSession

AFURLSessionManager创建并管理由 NSURLSessionConfiguration 设置过的 NSURLSession对象。该对象遵循<NSURLSessionDataDelegate>,
<NSURLSessionDownloadDelegate>,
<NSURLSessionDelegate>协议

  • 子类化注意事项

这个类是AFHTTPSessionManager的基类，AFHTTPSessionManager的优点在于优化了HTTP请求，如果想要扩展关于HTTP请求，请对AFHTTPSessionManager子类化。

  • 实现的代理方法

NSURLSessionDelegate

  • URLSession:didBecomeInvalidWithError:
  • URLSession:didReceiveChallenge:completionHandler:
  • URLSessionDidFinishEventsForBackgroundURLSession:

NSURLSessionTaskDelegate

  • URLSession:willPerformHTTPRedirection:newRequest:completionHandler:
  • URLSession:task:didReceiveChallenge:completionHandler:
  • URLSession:task:didSendBodyData:totalBytesSent:totalBytesExpectedToSend:
  • URLSession:task:didCompleteWithError:

NSURLSessionDataDelegate

  • URLSession:dataTask:didReceiveResponse:completionHandler:
  • URLSession:dataTask:didBecomeDownloadTask:
  • URLSession:dataTask:didReceiveData:
  • URLSession:dataTask:willCacheResponse:completionHandler:

NSURLSessionDownloadDelegate

  • URLSession:downloadTask:didFinishDownloadingToURL:
  • URLSession:downloadTask:didWriteData:totalBytesWritten:totalBytesWritten:totalBytesExpectedToWrite:
  • URLSession:downloadTask:didResumeAtOffset:expectedTotalBytes:

如果子类中有任何需要重载代理方法的，必须首先调用父类的代理方法的实现

附

NSURLSession

此API用来通过HTTP协议下载内容，提供了大量的代理方法来提供认证功能，使应用能够在后台或者当应用挂起的时候下载任务。

通过NSURLSession的API，应用创建了一系列的会话，每个都协调着一组相关的数据传输任务。比如，如果你在实现一个浏览器，那么应用会为每个标签或者窗口创建一个sesssion.每个session都会添加一系列的任务，每个任务都代表向一个特定的URL的网络请求。

一个会话的行为有一个设置对象决定，因为这里有三种类型的设置对象，所以相应的就有三种session对象。a.默认的与NSURLConnection很像，代表着session对象不缓存任何内容；b.download session把结果存储到一个文件中，并且当应用挂起、退出、甚至崩溃的时候持续性传输。

通过这些session对象，你可以调度三种类型的任务，数据存储在内存中的，下载到磁盘上的，将磁盘上的文件上传并接收响应的任务。

与其他网络API类似，NSURLSession也是异步的，它的数据返回有两种方式，当然，这取决于你调用了哪个方法:

  • 请求完成时的block，请求成功或者失败的时候通过block把数据返回到应用。
  • 接收到数据的时候调用自定义的代理方法。
  • 下载文件的时候调用自定义的代理方法。

上线发布检查清单（持续更新中）

在打包发布的时候有许多需要注意的细节，比如三方SDK的状态设置，日志的输出，编译标志的设置，繁复的细节如果能维护一份清单，那么会对以后的上线工作有利，也可以避免不必要的时间浪费.

  1. 应用更新功能是否可用
  2. 用户协议的网址
  3. 分享的连接是否可用
  4. 提交的审核信息中帐号是否可用
  5. 推送标记是否设置为发布模式。
  6. 提交版本的时候需要把ipa保存一份，以确保.dsYM文件能够保存，出现bug可以查看崩溃日志。
  7. 不要有log信息（important!）
